# 无人医院 AI 决策层 · 后端接入指南

本文面向**后端开发者**，说明如何把本仓库的 AI 决策层接入你们的服务。读完你应能：用真实大模型构造决策层、按一次就诊的状态机驱动它、拿到结构化的诊断/处置、并把每次诊疗的完整对话记录落盘。

- 模块路径：`medagent`，Go 1.22，**零外部依赖**（仅标准库）。
- 关键边界：**AI 决策层无状态、不做语义校验、不做编排**。状态、持久化、卡片/saga、支付、并发、熔断、真实编排都是**后端的职责**。本层只把"快照 → 结构化决策"这件事做好。

---

## 1. 模块总览

| 包 | 作用 | 后端是否直接用 |
| --- | --- | --- |
| `medagent/ai` | 决策层核心：4 个无状态 agent + typed intent 契约 + LLM 抽象 + 上下文构建 | ✅ 核心 |

## 2. 快速开始

```go
import (
    "context"
    "time"

    "medagent/ai"
    "medagent/ai/consultlog"
    "medagent/ai/openaicompat"
)

// 1) 构造真实 LLM 客户端（实现 ai.LLMClient）
llm := openaicompat.NewDeepSeek(os.Getenv("DEEPSEEK_API_KEY"), "deepseek-chat")
// 或任意 OpenAI 兼容端点（含第三方中转）：
// llm := openaicompat.New(openaicompat.Config{
//     BaseURL: "https://your-gateway/v1", APIKey: key, Model: "gpt-4o-mini",
// })

// 2)（推荐）包一层日志装饰器：本次就诊所有 LLM 调用按 visitID 落盘
llm = consultlog.Wrap(llm, consultlog.NewFileLogger("./logs"))

// 3) 构造决策层与急症守护
layer := ai.NewDecisionLayer(llm)
guardian := ai.NewGuardian(llm)

// 4) 每次就诊：在 ctx 上绑 visitID（让日志归档到本次就诊）
ctx := consultlog.WithVisitID(context.Background(), consultlog.NewVisitID())
ctx, cancel := context.WithTimeout(ctx, 60*time.Second)
defer cancel()

// 5) 由你的编排按状态机调用（见 §6）
snap := ai.Snapshot{Subjective: map[string]any{}}
snap.Interview = append(snap.Interview, ai.DialogTurn{Role: "patient", Text: "我发烧两天了"})
res, err := layer.Interview(ctx, snap)
// res.Reply 给患者；res.Advance != nil 表示问诊充分，可进入 triage
```

## 3. 核心接口

```go
// LLM 抽象：provider 中立的结构化输出。openaicompat 是其真实实现。
type LLMClient interface {
    Complete(ctx context.Context, req CompletionRequest) (CompletionResult, error)
}

// 主决策层：三段，全部无状态，每次从 Snapshot 重建上下文。
type DecisionLayer interface {
    Interview(ctx context.Context, s Snapshot) (InterviewResult, error)
    Triage(ctx context.Context, s Snapshot) (TriageDecision, error)
    Treatment(ctx context.Context, s Snapshot) (TreatmentPlan, error)
}
func NewDecisionLayer(llm LLMClient) *Layer

// 急症守护：纯判断，命中即建议打断转急诊。并发/ticker 由后端负责。
type Guardian interface {
    Assess(ctx context.Context, s Snapshot, ev Event) (EmergencyInterrupt, bool, error)
}
func NewGuardian(llm LLMClient) Guardian
```

> **无状态**：决策层不保存任何会话状态。每次调用你都要传入完整 `Snapshot`，AI 据此重建上下文（`buildMessages`：结构化字段全量带入，问诊仅保留最近 `InterviewRawTurns=6` 轮原文，更早折叠成摘要）。

## 4. 输入数据契约：`Snapshot`

`Snapshot` 是喂给 AI 的全量上下文，由后端组装：

```go
type Snapshot struct {
    Interview   []DialogTurn           // 问诊对话（患者/医生轮）
    Subjective  map[string]any         // 已采集的结构化主观信息（主诉/病史/体征）
    TestResults []TestResult           // 已回填的检验结果
    Diagnosis   *Diagnosis             // 已确诊（进入处置阶段后非空）
    PriorVisit  *VisitSummary          // 复诊时携带的上次就诊摘要（可为 nil）
    Refusals    []RefusalRecord        // 患者拒绝某项执行动作的记录
    Feedback    *OrchestratorFeedback  // 编排层瞬时反馈（见 §6.3）；非本次输入时置 nil
}

type DialogTurn struct { Role string; Text string } // Role: "patient" | "doctor"
type Diagnosis  struct { Name string; Basis string; Confidence float64 } // confidence 0~1
type Medication struct { Name, Dosage, Schedule string }
type TestResult struct { Item, Value string }
type VisitSummary struct { Diagnosis *Diagnosis; Medications []Medication }
type RefusalRecord struct { What string; At time.Time } // What: "test_pay" | "med_pay" ...
type Event struct { Kind string; Data any }             // Kind: "dialog" | "vital" | "test_result"

type OrchestratorFeedback struct {
    LastReject   RejectReason // 上次意图被你的语义校验拒绝的原因
    NextExpected string       // 你期望本次产出什么
    CardDeferred bool         // 上一张卡被暂缓
    MissingHint  []string     // 需补充的主观项（来自 triage 的 missing_subjective）
}
```

> **注意**：`Snapshot` 不含轮次计数——熔断（最大轮数）由**后端编排**强制，不提示给 AI。

## 5. 输出契约：typed intent

每段决策返回一个结构体，AI 已做**结构校验**（字段齐全 / enum 合法 / 自证字段存在）。你拿到的就是合法结构；**语义校验（状态合法性、能力是否具备、是否该熔断等）是你的职责**，通过 `OrchestratorFeedback` 回灌重决策。

### 5.1 Interview → `InterviewResult`

```go
type InterviewResult struct {
    Reply   string           // 总是给患者的话
    Advance *AdvanceToTriage // 非空 = 问诊充分，可进入 triage（此时 Reply 是过场话）
}
type AdvanceToTriage struct { Subjective map[string]any } // 本轮采集到的结构化主观信息
```

- `Advance == nil`：继续问诊。把 `Reply` 作为医生轮追加进 `Interview`，等患者下一句再调。
- `Advance != nil`：把 `Advance.Subjective` 合并进 `snap.Subjective`，进入 triage。

### 5.2 Triage → `TriageDecision`（三选一收敛）

```go
type TriageDecision struct {
    Decision            TriageChoice // "CONFIRM" | "INTERVIEW" | "TEST"
    Diagnosis           *Diagnosis   // CONFIRM 时非空
    MissingSubjective   []string     // INTERVIEW 时非空：还需采集的项
    SubjectiveExhausted bool         // TEST 时必须 true
    Reason              string       // TEST 时非空：为何必须检验
    TestItems           []string     // TEST 时非空：检验项目
}
const ( TriageConfirm TriageChoice = "CONFIRM"; TriageInterview = "INTERVIEW"; TriageTest = "TEST" )
```

- `CONFIRM`：把 `Diagnosis` 写进 `snap.Diagnosis`，进入处置。
- `INTERVIEW`：把 `MissingSubjective` 放进 `Feedback.MissingHint`，回到问诊补采。
- `TEST`：执行 `TestItems` 检验，把结果 append 进 `snap.TestResults`，再回 triage。

### 5.3 Treatment → `TreatmentPlan`（四选一处置）

```go
type TreatmentPlan struct {
    Plan               PlanKind     // "MEDICATION" | "TREATMENT" | "ADVICE_ONLY" | "REFERRAL"
    Advice             string       // 医嘱，恒非空
    Medications        []Medication // MEDICATION 时非空
    RequiredCapability string       // TREATMENT 时非空：用于比对本院能力清单的标识
    ReferralReason     string       // REFERRAL 时非空
}
const ( PlanMedication PlanKind = "MEDICATION"; PlanTreatment = "TREATMENT"; PlanAdviceOnly = "ADVICE_ONLY"; PlanReferral = "REFERRAL" )
```

- `TREATMENT`：用 `RequiredCapability` 比对本院能力清单。**不具备**时：设 `Feedback.LastReject = RejectCapabilityMissing` 回灌重决策（AI 通常改 REFERRAL）。

### 5.4 Guardian → `EmergencyInterrupt`

```go
type EmergencyInterrupt struct { Reason string }
// Assess 返回 (intent, hit, err)：hit==true 表示命中急症，建议立即打断转急诊。
```

### 5.5 语义校验回传原因 `RejectReason`

你在编排里做语义校验后，用它告诉 AI 上次为何被拒：

```go
const (
    RejectNone                   RejectReason = ""
    RejectSchemaInvalid          = "SCHEMA_INVALID"
    RejectIllegalTransition      = "ILLEGAL_TRANSITION"
    RejectCapabilityMissing      = "CAPABILITY_MISSING"
    RejectSubjectiveNotExhausted = "SUBJECTIVE_NOT_EXHAUSTED"
    RejectRoundLimitFused        = "ROUND_LIMIT_FUSED"
)
```

## 6. 一次就诊的编排（状态机）

> 真实编排（卡片/saga/持久化/支付/guardian 并发/熔断）是**后端职责**。`ai/harness.RunVisit` 是一份**最小参考实现**，可直接读它的源码理解主干，或起步时复用。

### 6.1 主干流程

```
问诊采集 ──► 收敛环 ──► 处置 ──► 终态
  ▲            │  │  │
  └─INTERVIEW──┘  │  └─TEST：回填检验结果后回到收敛环
                  └─CONFIRM：写入诊断，进入处置
```

1. **问诊阶段**：循环 `layer.Interview`，把患者消息与医生 `Reply` 交替追加进 `snap.Interview`，直到 `Advance != nil`，合并 `Subjective`。
2. **收敛环**：循环 `layer.Triage`：
   - `CONFIRM` → 写 `snap.Diagnosis`，进处置；
   - `INTERVIEW` → `snap.Feedback = &OrchestratorFeedback{MissingHint: td.MissingSubjective}`，回问诊；
   - `TEST` → 执行检验、`snap.TestResults = append(...)`，继续环。
3. **处置阶段**：`layer.Treatment`：`TREATMENT` 且能力不具备 → `Feedback.LastReject = RejectCapabilityMissing` 重决策；否则得终态（`ADVICE_ONLY`/`MEDICATION`/`REFERRAL`）。

### 6.2 参考编排 `harness.RunVisit`

```go
import "medagent/ai/harness"

out, err := harness.RunVisit(ctx, harness.Deps{
    Layer:       layer,                                  // ai.DecisionLayer
    Caps:        map[string]bool{"输液": true},           // 本院能力清单
    Patient:     func(lastDoctorReply string) string {…}, // 返回下一句患者消息
    TestResults: func(items []string) []ai.TestResult {…}, // 桩化检验回填
    Prior:       nil,                                     // 复诊上次摘要，可为 nil
})
// out: { Final "ADVICE"|"REFERRAL"; Diagnosis; Plan; Advice; Medications; Trace }
```

`RunVisit` 内置轮数护栏（问诊≤20、收敛≤10、处置≤5）防暴走——**生产里你应替换为自己的熔断策略**。它不驱动 guardian、无持久化，仅供参考/自测。

### 6.3 用 `OrchestratorFeedback` 回灌

AI 不记得上一轮。当你拒绝某个意图或需要它补采时，把原因放进 `snap.Feedback` 再调用；用完一次就置 `nil`（它是瞬时反馈，会渲染进【编排反馈】提示块）。

### 6.4 急症守护（并发）

`guardian.Assess(ctx, snap, ev)` 是**单次纯判断**。后端负责：开 goroutine/ticker 并发喂事件（`Event{Kind, Data}`），命中（`hit==true`）则抢占当前流程、结算、转急诊。守护不内部重试。

## 7. LLM adapter（`openaicompat`）

```go
type Config struct {
    BaseURL    string        // 形如 https://api.deepseek.com/v1
    APIKey     string
    Model      string
    Timeout    time.Duration // 0 → 默认 60s
    HTTPClient *http.Client  // 可选，便于注入/复用连接池
}
func New(cfg Config) *Client
func NewDeepSeek(apiKey, model string) *Client // 预填 BaseURL
func NewQwen(apiKey, model string) *Client     // 预填 DashScope 兼容模式 BaseURL
```

- **结构化输出走强制 tool-use**：adapter 把每段的 `OutputSchema` 映射为单一 function 并 `tool_choice` 强制调用，从 `tool_calls[0].arguments` 取结构化 JSON。
- **同角色消息合并**：adapter 会合并连续同角色消息（满足 Anthropic 风格 API 约束）。
- 预设 base URL：DeepSeek `https://api.deepseek.com/v1`；通义千问（DashScope 兼容模式）`https://dashscope.aliyuncs.com/compatible-mode/v1`。
- **选用的模型必须支持 function calling / tools**，否则强制 tool-use 会失败。
- **API key 由后端从环境/配置读取**，adapter 不读环境变量、不记录 key（key 在 HTTP 层，不进 `CompletionRequest`）。

四段对应的 tool（schema）名：`interview` / `triage_decide` / `treatment_plan` / `emergency_interrupt`——可用于日志里区分是哪个 agent。

## 8. 诊疗日志（`consultlog`）

```go
func Wrap(inner ai.LLMClient, sink Sink) *Client // 装饰器，返回 ai.LLMClient
func NewFileLogger(dir string) *FileLogger        // 每诊一文件 {dir}/{visitID}.jsonl
func NewVisitID() string                          // 形如 20260624-153012-a1b2c3d4
func WithVisitID(ctx, id string) context.Context  // 把 visitID 绑到 ctx
func VisitID(ctx) string                          // 取出 visitID
type Sink interface{ Write(rec CallRecord) error } // 自定义落点（DB/ELK）实现它即可
```

- 用法：`logged := consultlog.Wrap(real, consultlog.NewFileLogger("./logs"))`，并在每次就诊的 ctx 上 `WithVisitID`。该 ctx 下**所有** `Complete`（含 schema 重试、含你额外的患者模拟调用）都记入 `{dir}/{visitID}.jsonl`。
- **日志失败绝不影响诊疗**（sink 出错只写 stderr）。
- `CallRecord` 字段：`visit_id, time, schema, system, messages[{role,content}], structured, raw, latency_ms, error`。成功失败都记。
- **落点是后端的事**：默认给了文件 sink；接 DB/对象存储/ELK 自己实现 `Sink`。`/logs/` 建议加进 `.gitignore`（含医患对话，属医疗数据）。

## 9. 错误处理

```go
import "errors"

res, err := layer.Triage(ctx, snap)
switch {
case err == nil:
    // 用 res
case errors.Is(err, context.Canceled), errors.Is(err, context.DeadlineExceeded):
    // 被打断/超时：ctx 取消（含急症抢占）以原始 ctx 错误上抛，便于区分"被打断"与"LLM 故障"
case errors.Is(err, ai.ErrLLM):
    // 传输/超时/非 2xx/协议异常：LLM 调用失败，可重试或降级
default:
    var se *ai.SchemaError
    if errors.As(err, &se) {
        // 模型输出 K 次重试后仍不合 schema。se.Agent / se.Attempts / se.LastRaw / se.Cause
    }
}
```

- **`ai.ErrLLM`**：传输/超时/非 2xx/响应缺 tool_calls 等，统一包装，`errors.Is(err, ai.ErrLLM)` 成立。
- **`*ai.SchemaError`**：模型输出在包内自纠 `SchemaRetryMax=2` 次后仍不合 schema。
- **ctx 取消**：mid-flight 取消以**原始 ctx 错误**上抛（不归类为 `ErrLLM`），方便编排区分"被急症/超时打断"。
- adapter **不做网络重试**；需要退避重试时在外面包一层。

配置常量（`ai/config.go`）：`InterviewRawTurns = 6`（问诊保留原文轮数）、`SchemaRetryMax = 2`（schema 自纠次数）。

## 11. 接入边界与注意事项

- **AI 无状态**：每次传完整 `Snapshot`；会话状态/持久化/复诊数据由后端维护。
- **AI 不做语义校验、不做编排**：状态合法性、能力比对、轮数熔断、卡片/saga/支付、guardian 并发——都由后端实现；用 `OrchestratorFeedback` 把语义判定回灌给 AI。
- **`harness` 仅参考**：是 mock 编排，生产请实现自己的真实编排层。
- **provider 要求**：模型须支持 function calling；adapter 已处理同角色消息合并与强制 tool-use。
- **安全**：日志含医患对话（医疗数据），落点与脱敏由后端决定；API key 不入日志。
- **并发**：`DecisionLayer`/`Guardian`/`consultlog` 均可并发使用（FileLogger 内部加锁）；同一就诊的 LLM 调用建议串行驱动，guardian 可与主流程并发。

## 12. 一段最小可跑的编排骨架（不依赖 harness）

```go
func runVisit(ctx context.Context, layer ai.DecisionLayer, caps map[string]bool,
    nextPatientMsg func(lastDoctorReply string) (string, bool), // bool=false 表示患者结束
    runTests func(items []string) []ai.TestResult,
) (ai.TreatmentPlan, *ai.Diagnosis, error) {

    snap := ai.Snapshot{Subjective: map[string]any{}}

    interview := func() error {
        reply := ""
        for i := 0; i < 20; i++ { // 后端自己的熔断
            msg, ok := nextPatientMsg(reply)
            if !ok { return fmt.Errorf("患者提前结束") }
            snap.Interview = append(snap.Interview, ai.DialogTurn{Role: "patient", Text: msg})
            res, err := layer.Interview(ctx, snap)
            if err != nil { return err }
            if res.Advance != nil {
                for k, v := range res.Advance.Subjective { snap.Subjective[k] = v }
                snap.Feedback = nil
                return nil
            }
            snap.Interview = append(snap.Interview, ai.DialogTurn{Role: "doctor", Text: res.Reply})
            reply = res.Reply
        }
        return fmt.Errorf("问诊未收敛")
    }

    if err := interview(); err != nil { return ai.TreatmentPlan{}, nil, err }

    for r := 0; r < 10; r++ { // 收敛环熔断
        td, err := layer.Triage(ctx, snap)
        if err != nil { return ai.TreatmentPlan{}, nil, err }
        switch td.Decision {
        case ai.TriageConfirm:
            snap.Diagnosis = td.Diagnosis
            snap.Feedback = nil
            for t := 0; t < 5; t++ { // 处置环熔断
                tp, err := layer.Treatment(ctx, snap)
                if err != nil { return ai.TreatmentPlan{}, nil, err }
                if tp.Plan == ai.PlanTreatment && !caps[tp.RequiredCapability] {
                    snap.Feedback = &ai.OrchestratorFeedback{LastReject: ai.RejectCapabilityMissing}
                    continue
                }
                return tp, snap.Diagnosis, nil
            }
            return ai.TreatmentPlan{}, nil, fmt.Errorf("处置未收敛")
        case ai.TriageInterview:
            snap.Feedback = &ai.OrchestratorFeedback{MissingHint: td.MissingSubjective}
            if err := interview(); err != nil { return ai.TreatmentPlan{}, nil, err }
            snap.Feedback = nil
        case ai.TriageTest:
            snap.TestResults = append(snap.TestResults, runTests(td.TestItems)...)
        }
    }
    return ai.TreatmentPlan{}, nil, fmt.Errorf("收敛环未收敛")
}
```

把 `nextPatientMsg`/`runTests` 换成你的真实终端交互与检验子系统，再叠加 guardian 并发、卡片/支付、持久化与熔断，即是生产编排。
